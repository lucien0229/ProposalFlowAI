# 任务01-单仓脚手架

## 1. 任务目标

初始化 ProposalFlow AI 的 monorepo 工作区，建立稳定的顶层结构、四个应用边界与共享 packages 承载位，为后续阶段提供不会被推翻的工程基础。

## 2. 参考文档

- `docs/share/ProposalFlow AI 开发骨架与目录结构设计说明 v1.0.md`
- `docs/share/ProposalFlow AI｜系统架构与技术方案 v1.0.md`
- `docs/share/ProposalFlow AI｜共享平台边界说明 v1.0.md`

## 3. 前置条件

- 仓库当前仍以文档为主，尚未存在正式工程骨架
- 已确认采用单仓 monorepo、`apps/web` + `apps/admin` + `apps/api` + `apps/worker`

## 4. 主要工作内容

- 初始化 workspace 根配置（pnpm workspace / package manager 基线）
- 创建顶层目录：`apps/`、`packages/`、`infra/`、`scripts/`、`tests/`
- 创建四个应用目录及最小入口：
  - `apps/web`
  - `apps/admin`
  - `apps/api`
  - `apps/worker`
- 创建共享包目录及最小导出：
  - `packages/shared-types`
  - `packages/shared-config`
  - `packages/shared-schemas`
- 约束 web/admin/api/worker 的职责边界，不在此 task 中实现业务能力
- 在 API 骨架中预留 product/admin namespace 与 guard 占位：`/api/v1/*`、`/api/v1/admin/*`、workspace/admin guard 分离的目录与入口

## 5. 交付物

- workspace 根配置文件
- 四个 app 的最小可运行脚手架
- 三个 shared packages 的最小结构与导出文件
- 顶层目录结构与基础启动脚本入口
- API 中 `product` / `admin` 路由目录与 guard 占位骨架

## 6. 验收标准

- 能在仓库根目录完成依赖安装
- `apps/web`、`apps/admin`、`apps/api`、`apps/worker` 各自都有最小启动入口
- shared packages 能被至少一个 app 成功引用
- 工程结构符合共享文档中的边界约束
- `apps/api` 内已存在 `/api/v1/*` 与 `/api/v1/admin/*` 的 namespace 骨架与 guard 占位，不需要到阶段 1 再回头重构目录

## 7. 风险与边界

- 不在本 task 内引入过早抽象（如 `packages/ui`）
- 不把 admin 页面或 admin route 混入 web app
- 不在此 task 内实现数据库、鉴权、业务路由

## 8. 与上下游任务关系

- 为 `任务02-本地开发基线.md`、`任务03-最小CI基线.md`、阶段 1 全部任务提供目录基础
