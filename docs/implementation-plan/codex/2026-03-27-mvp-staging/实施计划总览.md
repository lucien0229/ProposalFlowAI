# ProposalFlow AI — Codex 实施计划总览（MVP → Staging）

**目标：**在 monorepo 内交付 web/admin/api/worker 的 MVP 能力，并在 **Staging（VM + Compose）** 上完成可重复部署与验收。

## Source of truth

- 产品与技术规格：见 `docs/web/`、`docs/admin/`、`docs/share/` 下已冻结文档
- Codex 正式执行计划：`docs/implementation-plan/codex/2026-03-27-mvp-staging/`
- Claude 正式设计计划：`docs/implementation-plan/claude/2026-03-27-mvp-staging/`

## 角色边界

- 本目录仅承载：开发、实现、测试、部署、staging 验收
- 布局、导航、关键页面线框、`.pen` 资产与设计冻结规则由 Claude 目录承载
- 当前端任务依赖页面骨架时，以 Claude 目录中的冻结资产为准

## 阶段划分

- 阶段 0：Monorepo Scaffold（工程骨架）
- 阶段 1：Shared Platform（底座）
- 阶段 2：Web P0（用户侧闭环）
- 阶段 3：Admin P0（运营端闭环）
- 阶段 4：Staging 上线（VM + Compose）

## 开发侧前置依赖

- Web/Admin 页面实现前，应先引用 Claude 侧冻结的：
  - 导航结构
  - Layout 骨架
  - 页面与路由映射
  - 状态位与基础结构说明
- 阶段 2/3 的页面开发开始前，必须同时满足：
  - Codex 阶段 0（工程骨架）完成
  - Claude 阶段 0（设计冻结）完成
- staging 验收以已冻结的 Claude 设计资产为页面结构基线；不在阶段 4 临时改动 IA、导航层级或整体布局方向

## 验收口径

- Web 主流程可跑通（Auth → Setup → Opportunity → Lead Brief → Discovery → Proposal → Follow-up）
- Admin P0 五模块可用（Overview / Workspaces / Users / Subscriptions & Trials / Funnel）
- Shared Platform 底座可验证
- 镜像、compose、迁移、seed、smoke tests、回滚路径齐全
