# 00-总控计划（Codex｜MVP → Staging）

## 1. 目标

在不推翻工程骨架的前提下，完成 ProposalFlow AI 的开发、测试、部署与 staging 验收交付。

## 2. 阶段划分

- 阶段 0：Monorepo Scaffold（工程骨架）
- 阶段 1：Shared Platform（底座）
- 阶段 2：Web P0（用户侧闭环）
- 阶段 3：Admin P0（运营端闭环）
- 阶段 4：Staging 上线（VM + Compose）

## 3. 全局约束（冻结）

- web/admin 双会话分离：`session_type = web | admin`
- API namespace 分离：`/api/v1/*` vs `/api/v1/admin/*`
- admin 只读优先，不做高风险写操作
- 指标与漏斗口径：workspace-level funnel 起点 `workspace_created`；`workflow_completion_rate` 分母为 Activated Workspaces
- 计费受限状态统一阻断：generate、regenerate、save current、save-version、restore、export
- staging：VM 上 compose 仅编排 web/admin/api/worker；数据库/Redis/对象存储为独立 staging 资源

## 4. 设计依赖

- 前端实现任务默认依赖 Claude 目录中已冻结的设计资产：
  - `docs/implementation-plan/claude/2026-03-27-mvp-staging/阶段0/任务04-设计与界面基础.md`
  - `docs/implementation-plan/claude/2026-03-27-mvp-staging/规范/设计交接标准.md`
  - `docs/implementation-plan/claude/2026-03-27-mvp-staging/规范/设计冻结规则.md`
- 页面实现开始前，必须同时满足：
  - Codex 阶段 0 完成（工程骨架、本地基线、最小 CI）
  - Claude 阶段 0 完成（导航、布局骨架、关键页面线框冻结）
- staging 验收继续以 Claude 冻结资产为页面结构基线，不在阶段 4 临时改动 IA、导航或整体布局方向

## 5. 全局验收（staging）

- [ ] Web 主流程可跑通（Auth → Setup → Opportunity → Lead Brief → Discovery → Proposal → Follow-up）
- [ ] Admin P0 五模块可用（Overview / Workspaces / Users / Subscriptions / Trials / Funnel）
- [ ] 平台底座可验证（双会话、billing 回写与限制解释、PDF 状态与 retry、AI 生成/失败、activity_logs 可聚合）
- [ ] 部署可重复（镜像 + compose + 迁移 + seed + smoke tests + 回滚路径）
